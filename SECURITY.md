# Security

This document describes the security model of the AI-Powered RPA System: what
the safety engine blocks by default, how to override it when you genuinely need
to, the protections built into path handling and audit logging, and how to
report a vulnerability.

The RPA system drives real mouse, keyboard, and application actions on your
machine, and it is designed to consume workflows that may be generated by an
LLM or another untrusted source. Because of that, the system is **secure by
default**: it statically inspects every workflow before executing it and
**refuses to run anything that contains a CRITICAL finding** unless you
explicitly opt out.

---

## Threat model

The system assumes that a workflow may originate from a source you do not fully
trust — most importantly, an LLM that turned a natural-language prompt into a
sequence of actions, or a JSON file authored elsewhere. The primary risks it
defends against are:

- **Destructive command execution.** A workflow step whose text/target/notes is
  shaped like a real, destructive shell command (recursive force-deletes, disk
  formatting, registry destruction, fork bombs, `curl | sh` remote-code
  execution, database `DROP`/`TRUNCATE`, etc.).
- **Shell/script egress.** An `open_application` / `close_application` step that
  launches a shell or scripting host (`cmd`, `powershell`, `bash`, `wscript`,
  `mshta`, `rundll32`, `regedit`, …), which would let an attacker run arbitrary
  commands outside the RPA action vocabulary.
- **Resource exhaustion / denial of service.** Pathologically long workflows or
  absurd wait/timeout values that would hang the machine.
- **Path traversal.** A `screenshot` step whose desired filename tries to escape
  the screenshot directory (absolute paths, `..`, drive letters) to overwrite
  files elsewhere.
- **Secret leakage.** Passwords and other sensitive values being written to logs
  or audit trails in clear text.

The system does **not** sandbox the GUI backend itself, and it cannot reason
about the *semantics* of arbitrary coordinate clicks. The scanner is a static,
GUI-free defense layer; it is a strong default, not a guarantee against every
possible misuse. Treat every workflow you did not author as untrusted, prefer
`--dry-run` first, and keep `--unsafe` for trusted inputs only.

## What the safety engine blocks by default (CRITICAL only)

Before execution, `WorkflowExecutor.execute_workflow` runs
`scan_workflow(workflow, policy)` (see `ai_rpa_system/security.py`). The scan is
pure-Python and headless-safe (no `pyautogui`/`pynput` imports). It returns a
list of `SecurityFinding`s at three severities: **critical**, **major**, and
**minor**.

**Only CRITICAL findings block execution.** When a scan produces one or more
critical findings, the workflow is refused: a failed `ExecutionResult` is
returned with the critical findings as `errors` and **no step is executed**.
MAJOR and MINOR findings never block — they are surfaced as `warnings` on the
result so you can review them without halting automation.

The design is intentionally **narrow about what counts as CRITICAL** so that
ordinary prose (e.g. "please delete the old file") does not trip the scanner —
only text shaped like a real command does. The critical categories are:

| Category | What triggers it |
| --- | --- |
| `destructive_command` | A `text`, `target`, `notes`, or `description` field matches a command-shaped destructive pattern (see below). |
| `shell_launch` | An `open_application` / `close_application` target resolves to a blocked shell/script host. |
| `workflow_too_long` | The workflow has more steps than `max_steps` (default **200**). |
| `excessive_wait` | A `wait_before`, `wait_after`, `timeout`, or `poll_interval` exceeds `max_wait_seconds` (default **300s**). |

Destructive command patterns (matched case-insensitively, anchored to command
syntax) include:

- POSIX recursive/forced removal: `rm -rf /`, `rm -rf ~`, `rm -rf *`, `rm -rf $HOME`
- Windows force/quiet/recursive delete: `del /f`, `del /q`, `erase /f`, `rmdir /s`, `rd /s`
- PowerShell: `Remove-Item ... -Recurse`, `Remove-Item ... -Force`, `ri ... -Recurse`
- Disk/filesystem destruction: `format c:`, `mkfs`, `diskpart`, `cipher /w`, `dd if=... of=/dev/...`, `> /dev/sdX`, `fdisk`, `wipefs`, `shred`
- System power state: `shutdown`, `reboot`, `halt`, `poweroff`, `init 0`
- Registry destruction: `reg delete`, `Remove-Item HKLM:` (and other hives)
- Fork bomb: `:(){ ... }`
- Ownership/permission wipes: `chmod -R 000`, `chown -R ... /`
- Recursive delete via find: `find / ... -delete`, `find ... -exec rm`
- Remote code execution: `curl ... | sh`, `wget ... | bash` (and `sudo`/`pwsh`/`powershell` variants)
- Database destruction: `DROP DATABASE`, `DROP TABLE`, `TRUNCATE TABLE`

Blocked shell/script launch targets (matched token-aware on the basename, so
`notepad` does not match `reg` and `shell` does not match `sh`): `cmd`,
`cmd.exe`, `powershell`, `pwsh`, `bash`, `sh`, `zsh`, `regedit`, `reg`,
`wscript`, `cscript`, `mshta`, `rundll32`.

### Non-blocking findings (for awareness)

- **MAJOR** — `suspicious_coordinates` (negative coordinates),
  `oversized_text` (text longer than `max_text_length`, default **5000**),
  `risky_hotkey` (`alt` combined with `del`/`f4`).
- **MINOR** — `style` (a step with no human-readable description).

## How to override (`allow_unsafe` / `--unsafe`)

The block exists to protect you; override it only for inputs you trust and have
reviewed.

**Library:** pass `allow_unsafe=True` to `execute_workflow`. This **skips the
scan entirely** and runs the workflow even if it would otherwise contain
critical findings:

```python
from ai_rpa_system import WorkflowExecutor

executor = WorkflowExecutor()
result = executor.execute_workflow(workflow, allow_unsafe=True)
```

You can also keep the scan on but tune the thresholds/blocklists with a custom
policy instead of disabling it:

```python
from ai_rpa_system import WorkflowExecutor, SafetyPolicy

policy = SafetyPolicy(max_steps=500, max_wait_seconds=600)
result = executor.execute_workflow(workflow, policy=policy)
```

To leave the scan enabled (default), simply call `execute_workflow(workflow)` —
`safe=True` by default.

**CLI:** pass `--unsafe` to `run` or `run-prompt`:

```bash
ai-rpa run my_workflow --unsafe
ai-rpa run-prompt "..." --unsafe
```

To **inspect** findings without running anything, use the read-only scanner:

```bash
ai-rpa scan my_workflow
ai-rpa scan --file ./workflow.json
```

`ai-rpa scan` prints findings grouped by severity and exits non-zero only when a
CRITICAL finding exists.

## Path-traversal protection

Screenshot output is confined to the executor's `screenshot_dir`. A
`screenshot` step's `notes` field is treated as a desired filename and is
sanitized before use: absolute paths, leading `/` or `\`, parent-directory
traversal (`..`), and drive-letter/colon forms (`C:`) are all rejected, and the
resolved path is verified to remain *under* the screenshot directory. Anything
that would escape is silently replaced with an auto-generated safe name inside
`screenshot_dir`. This prevents a malicious workflow from overwriting arbitrary
files via the screenshot path.

## Injection protection

- **No implicit command execution.** The RPA action vocabulary is a fixed set of
  GUI primitives (click, type, hotkey, scroll, etc.). The system never passes
  step text to a shell. The only path that can launch an external program is
  `open_application` / `close_application`, and that path is guarded by the
  `shell_launch` blocklist above.
- **Command-shaped text is caught statically.** Even though step text is not
  executed as a shell command, destructive command strings are flagged as
  CRITICAL so that a workflow which *expects* an app to interpret them (e.g. a
  terminal already in focus) is blocked before any keystrokes are sent.
- **Variable substitution is literal.** `{{variable}}` placeholders are replaced
  with their literal string values via a fixed `\{\{(\w+)\}\}` pattern; unknown
  placeholders are left untouched, and substitution does not evaluate or
  interpret the substituted content.

## Secret redaction

Mark any step that types a credential or other secret with `sensitive=True`:

```python
ActionStep(action="type", text="{{password}}", sensitive=True,
           description="Enter password")
```

When a step is `sensitive`:

- The automation engine's `type_text` is called with `sensitive=True`, so the
  backend **never logs the raw text** — it logs a redacted placeholder with only
  the length.
- The audit log (see below) records the step as `"text": "[REDACTED]"` instead
  of any value.

The **audit log never writes raw step text at all**, sensitive or not. Each
executed step appends one JSON line with keys `{step, action, success, dry_run,
ts_monotonic}` (plus `"text": "[REDACTED]"` for sensitive steps). Audit-log I/O
failures are swallowed so they can never break or alter a run. Enable it with
`audit_log=...` (library) or `--audit PATH` (CLI):

```bash
ai-rpa run my_workflow --audit ./run.jsonl
```

## Responsible use

This tool performs **real mouse, keyboard, and application actions on your
machine**. Use it responsibly:

- Only run workflows you trust or have reviewed. Prefer `--dry-run` first to see
  what a workflow would do without side effects.
- The CLI **will not perform a real (non-dry) run without confirmation**: in an
  interactive terminal it prompts once, and in a non-interactive/piped context
  it refuses unless you pass `--yes`.
- Keep `--unsafe` / `allow_unsafe=True` for trusted, reviewed inputs only.
- Do not use this software to access systems you are not authorized to use, to
  bypass access controls, or for any unlawful purpose. You are responsible for
  how you use it.

## Reporting a security issue

If you discover a vulnerability, please report it **privately** rather than
opening a public issue. Email `valentinohelp@gmail.com` with:

- A description of the issue and its impact.
- Steps to reproduce (a minimal workflow JSON is ideal).
- The version (`ai-rpa --version`) and platform.

Please give us a reasonable window to investigate and ship a fix before any
public disclosure. We appreciate responsible disclosure.
