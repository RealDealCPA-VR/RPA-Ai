## Core Components

### 1. Data Models (`models.py`)

**Purpose**: Define the data structures used throughout the system.

**Key Classes**:
- `ActionStep`: Single automation action
- `Workflow`: Complete automation workflow
- `PromptAction`: Parsed natural language prompt
- `ExecutionResult`: Workflow execution result

**Design Philosophy**: 
- LLM-friendly JSON serialization
- Pydantic validation for data integrity
- Clear, descriptive field names
- Comprehensive metadata

### 2. Automation Engine (`automation_engine.py`)

**Purpose**: Low-level desktop automation capabilities.

**Key Features**:
- Cross-platform support (Windows, macOS, Linux)
- Mouse control (click, move, drag, scroll)
- Keyboard control (type, press, hotkeys)
- Screen capture and image recognition
- Coordinate-based and image-based element detection

**Technology**: Built on PyAutoGUI for cross-platform compatibility.

### 3. Prompt Parser (`prompt_parser.py`)

**Purpose**: Convert natural language into structured automation actions.

**Key Features**:
- Pattern-based intent detection
- Entity extraction (URLs, applications, text)
- Multi-step prompt parsing
- Confidence scoring
- LLM-friendly output format

**Supported Patterns**:
- Click actions
- Type actions
- Navigation
- Application launching
- Key presses and hotkeys
- Wait commands

### 4. Workflow Manager (`workflow_manager.py`)

**Purpose**: Manage workflow lifecycle (record, save, load, organize).

**Key Components**:
- `WorkflowRecorder`: Records user actions into workflows
- `WorkflowManager`: Handles workflow storage and retrieval

**Features**:
- Real-time action recording
- JSON-based storage
- Workflow listing and search
- LLM-friendly export format
- Metadata management

### 5. Executor (`executor.py`)

**Purpose**: Execute workflows with full LLM integration support.

**Key Features**:
- Workflow execution with variable substitution
- Natural language prompt execution
- Error handling and retry logic
- Screenshot capture during execution
- Detailed execution reporting
- LLM-friendly result formatting

## Data Flow

### Workflow Creation Flow

```
User Action → Recorder → ActionStep → Workflow → JSON File
```

### Prompt Execution Flow

```
Natural Language Prompt → Parser → PromptAction → Workflow → Executor → Result
```

### Saved Workflow Execution Flow

```
Workflow Name → Manager → Workflow → Executor → Result
```

### LLM Integration Flow

```
LLM Prompt → JSON Workflow → Parser → Workflow → Executor → Result → LLM Analysis
```

## Key Design Principles

### 1. LLM-First Design

Every component is designed to be easily consumed and generated by LLMs:
- Structured JSON formats
- Clear, descriptive field names
- Human-readable descriptions
- Comprehensive metadata
- Validation with helpful error messages

### 2. Modularity

Each component is independent and can be used separately:
- Use `AutomationEngine` directly for low-level control
- Use `PromptParser` for natural language processing
- Use `WorkflowManager` for workflow organization
- Use `Executor` for execution only

### 3. Cross-Platform Compatibility

Works on Windows, macOS, and Linux:
- Platform detection
- OS-specific key mappings
- Screen resolution handling
- Path normalization

### 4. Error Resilience

Comprehensive error handling:
- Try-catch blocks around all operations
- Detailed error messages
- Retry logic for unreliable actions
- Graceful degradation

### 5. Extensibility

Easy to extend with new features:
- Plugin-style action types
- Custom prompt patterns
- Additional storage backends
- Custom execution strategies

## File Formats

### Workflow JSON Format

```json
{
  "name": "workflow_name",
  "description": "What this workflow does",
  "steps": [
    {
      "action": "click",
      "description": "Human-readable description",
      "coordinates": [x, y],
      "wait_before": 0.5,
      "wait_after": 1.0
    }
  ],
  "variables": {
    "key": "value"
  },
  "created_at": "2024-01-01T00:00:00",
  "updated_at": "2024-01-01T00:00:00",
  "tags": ["tag1", "tag2"],
  "retry_on_failure": true,
  "max_retries": 3
}
```

### Execution Result Format

```json
{
  "workflow_name": "name",
  "success": true,
  "steps_completed": 5,
  "total_steps": 5,
  "execution_time": 10.5,
  "errors": [],
  "warnings": [],
  "step_results": [...],
  "screenshots": ["path1.png", "path2.png"],
  "timestamp": "2024-01-01T00:00:00"
}
```

## Extension Points

### Adding New Action Types

1. Add action type to `ActionStep` model
2. Implement execution logic in `Executor._execute_step()`
3. Add parsing pattern to `PromptParser`
4. Update documentation

### Adding New Prompt Patterns

1. Add pattern to `PromptParser.action_patterns`
2. Implement step generation in `PromptParser._generate_steps()`
3. Test with various phrasings

### Custom Storage Backends

1. Extend `WorkflowManager`
2. Implement `save_workflow()` and `load_workflow()`
3. Maintain JSON compatibility

### LLM Integration Patterns

1. Create integration class
2. Implement prompt generation
3. Add result parsing
4. Handle feedback loops

## Performance Considerations

### Execution Speed
- Default pause: 0.5s between actions
- Configurable wait times per step
- Parallel execution not supported (sequential only)

### Memory Usage
- Workflows stored as JSON (minimal memory)
- Screenshots can consume significant disk space
- Recorder stores actions in memory during recording

### Scalability
- Suitable for workflows up to 100 steps
- No built-in workflow scheduling (use external scheduler)
- Single-threaded execution

## Security Considerations

### Input Validation
- All inputs validated with Pydantic
- Coordinate bounds checking
- File path sanitization

### Execution Safety
- PyAutoGUI failsafe enabled (move to corner to abort)
- No automatic privilege escalation
- User confirmation for destructive actions (recommended)

### Data Storage
- Workflows stored as plain JSON
- No encryption by default
- Sensitive data should use variables, not hardcoded values
